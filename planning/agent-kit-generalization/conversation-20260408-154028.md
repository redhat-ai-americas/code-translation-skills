# Conversation — 2026-04-08

Ideation session on generalizing code-translation-skills from py2to3-specific to a language-agnostic agent kit.

## How the session started

User described the current state: the skill suite worked well on a real Python 2→3 engagement, then the same engagement switched to Java 8→17 upgrade work and skills had to be rewritten. That's unsustainable across many languages. User pointed at five PyPI packages they maintain (veripak, treeloom, greploom, stigcode, sanicode) and asked how they fit in.

User also raised the question of whether to run anything as MCP on OpenShift, with the caveat that it shouldn't add weight without benefit.

User's reference point: Claude Code agent loops, not LangGraph or other workflow engines. And they flagged that coding agents can't just be pointed at a 20-year-old codebase and asked to refactor — there needs to be a prescribed process.

## What I researched

Pulled package descriptions from PyPI JSON API for all five packages. Read the current project's README, SKILL-UPDATE-PLAN.md (completed rounds 1-7), and BACKLOG.md.

Key findings:
- treeloom is a mature multi-language code property graph library (Python, JS/TS, Go, Java, C, C++, Rust) with taint analysis — directly replaces the `universal-code-graph` skill internals
- greploom is the context retrieval piece I didn't know was missing. Hybrid vector+BM25 semantic search over treeloom CPGs with graph-aware neighborhood expansion and LLM token budgets. Has MCP serve mode.
- veripak is multi-ecosystem dependency health auditing with migration difficulty assessment
- sanicode is SAST with compliance framework mapping (Python/JS/PHP only — no Java yet, that's a Phase 1 gap)
- stigcode is SARIF → DISA STIG/NIST 800-53 compliance artifacts

Three of the five are already multi-language, which is where the leverage is.

## Key insight from research

The current skill suite replicates (imperfectly) what treeloom + greploom already solve. The generalization isn't "port the Python scripts to other languages" — it's "stop replicating these packages and delegate to them." The skills become structured prompts around package CLIs.

## The reframe on Mode B

User wants both transform migration (what exists) and spec-driven rewrite (new). The current behavioral-contract-extractor works at function level, which isn't enough for a rewrite workflow. Needed a spec hierarchy: function → module → subsystem → system, each level built bottom-up from the one below plus LLM reasoning. Each level serves a different purpose (function contracts = test oracle, module specs = generation target, subsystem specs = architectural decisions, system spec = stakeholder artifact).

Flagged the chicken-and-egg risk: if the spec extractor mirrors existing (bad) module boundaries, the rewrite inherits the mess. Asked whether to mirror or propose alternatives.

## Key decisions reached

User's answers to framing questions:

1. **Scope:** Both Java unblock and broad kit, phased. Java first.
2. **Mode priority:** Equal weight Mode A and Mode B. This was the more ambitious choice.
3. **Existing skills:** Rename and parameterize in place. Preserve git history.
4. **greploom deployment:** Dual mode — CLI default, MCP on OpenShift optional.
5. **Java phases needed:** Discovery, Mechanical, Verification, Cutover. This revealed that the 6-phase backbone is really a 6-phase *max*, and profiles should declare which phases are active.
6. **Subsystem boundaries:** Propose alternatives, not mirror. This is the big differentiator.
7. **Mode B dashboard:** Parked.
8. **Capture notes:** Yes.

## What I proposed

Two-phase roadmap:

**Phase 1 (Java unblock):** Minimal parameterization to get Java 8→17 moving. Rename skills. Extract Python-specific bits into `profiles/python-2-to-3/`. Author `profiles/java-8-to-17/` in parallel. Wire in veripak and stigcode (both work today). Defer treeloom/greploom/sanicode internal delegation to Phase 2. Handle Java AST transforms via OpenRewrite or tree-sitter. Use Semgrep for Java SAST until sanicode adds Java.

**Phase 2 (Kit generalization):** Swap foundation tools (treeloom/greploom/sanicode). ## The reframe (later in the same session)

After the initial Phase 1 / Phase 2 roadmap was proposed, the user came back with a significant reframe that changed the direction of the design:

> "I'm less concerned about sequencing for this project because it's not in active use. This was a first try effort that worked great for one Python project, but then couldn't scale to other codebases, and even on Python seemed to be trying to make an agent loop perform like a deterministic agentic workflow. Some great nuggets are in those skills and documents, but the approach was wrong. It would be worth preserving a version of this in a peer repo just for posterity in case we don't produce anything here that's any better, but I don't want the past to be the enemy of the future, which can happen if we try to be too incremental sometimes."

This is the key insight: **the old project was trying to make an agent loop behave like a deterministic workflow.** Every round of fixes (phase runners, gate checkers, TODO templates) was adding rails to force the agent through a prescribed process. The agent kept drifting, so the rails kept growing. That's fighting the agent instead of leveraging it.

The user also flagged that "propose alternative decompositions" is the killer feature — the thing that will generate demand for this kit — and it deserves its own dedicated research session.

### What this changed

The Phase 1 (rename + parameterize in place) / Phase 2 (refactor) roadmap became obsolete. Replaced with a fresh-start approach:

1. Archive the current `code-translation-skills` project as posterity in a peer repo
2. Start a new repo with an agent-loop-native design from day one
3. Draw good nuggets forward deliberately (domain insights, reference docs, behavioral contract concept, dashboard design) but don't carry the orchestration machinery
4. Build fewer, larger skills (~10-15) instead of 34 fine-grained ones
5. Skills describe intent and verification; the agent loop handles sequencing
6. No phase runners, no blocking gate checkers

## OpenRewrite research

User asked to research OpenRewrite specifically before wrapping up. General-purpose research agent returned a detailed findings report (captured in `openrewrite-findings.md`).

**Key findings:**
- OpenRewrite is the right answer for Java mechanical conversion — mature, battle-tested meta-recipes (`UpgradeToJava17`, `UpgradeToJava21`, `JavaxMigrationToJakarta`), used at scale by AWS, Spring, Dropwizard
- Covers most Java 8→17 migration patterns including var, text blocks, switch expressions, pattern matching, jakarta namespace, deprecated API removal
- Gaps: sealed class conversion, `ClassToRecord` (verify status), `module-info.java` generation — these stay in the semantic/LLM-driven phase
- Invocation: Maven plugin in `dryRun` mode produces a unified diff patch and CSV data tables — trivially parseable by an agent
- No native JSON or SARIF output — build a thin adapter (~50 lines) that converts patch + CSV to normalized JSON
- **Does NOT solve Python 2→3.** `rewrite-python` was archived in January 2026. Use libcst for Python mechanical work.
- **Not a universal engine.** OpenRewrite is the Java engine, full stop. Each other language needs its own adapter (libcst for Python, jscodeshift/ts-morph for JS/TS, Roslyn for C#, etc.)
- Licensing gotcha: `rewrite-migrate-java` is a mixed Apache 2.0 / Moderne Source Available License (MSAL) repo. Running recipes is fine under either. Bundling recipes in a commercial product needs legal review.
- Biggest failure mode: type attribution silently fails if classpath is broken, recipes quietly match nothing. Agent must pre-check `mvn compile` passes.

**Decision:** Adopt OpenRewrite as the Java mechanical engine. Build a thin JSON adapter. Do not plan to use it for any other language.
